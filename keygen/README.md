# keygen

Generates the two RSA-2048 keypairs fota-secure needs. Run once per
environment (dev, CI, or a real provisioning workflow) — never share a
single keypair's private key outside where it's meant to live.

## Usage

Use a virtualenv rather than installing `cryptography` into your global/
system Python — this also gets you editor support (import resolution,
type checking) for `generate_keys.py`, which will otherwise show
unresolved-import warnings if `cryptography` isn't installed anywhere your
editor can see it.

```
python3 -m venv .venv
source .venv/bin/activate  # .venv\Scripts\activate on Windows
pip install -r keygen/requirements.txt
python3 keygen/generate_keys.py --output-dir ./keys
```

`.venv/` is already covered by this repo's `.gitignore`.

Produces, in `--output-dir` (default `./keys`, already gitignored):

- `signing_private.pem` / `signing_public.pem`
- `device_private.pem` / `device_public.pem`

## What each keypair is for

- **Signing keypair** — the private key signs release packages (see
  `docs/FORMAT_SPEC.md`'s Signature Section); it should live only wherever
  releases are authorized from (offline, a secrets manager, or an HSM in a
  real deployment), never on the device. The public key is baked into the
  consumer binary at provisioning time to verify signatures.
- **Device encryption keypair** — the public key wraps the fresh AES key
  generated for each build (RSA-OAEP envelope encryption); the private key
  lives only on the device, so it can unwrap that key at install time.

These are deliberately two separate keypairs, not one reused for both
roles — see `docs/THREAT_MODEL.md` item 1 for why.

## Never commit private keys

**`*_private.pem` files must never be committed to source control.** This
repo's `.gitignore` excludes `*.pem` and `*.key` for exactly this reason,
but that's a backstop, not a substitute for care — see
`docs/THREAT_MODEL.md` item 2: a committed private key means every
device's security depends on the access-control history of a git
repository forever, which isn't defensible. Keys generated by this script
are for local development and testing only, unless you've deliberately
wired them into a real secrets-management flow.
